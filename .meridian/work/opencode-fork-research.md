# OpenCode --fork: Research Notes

Date: 2026-03-29

## Which OpenCode?

There are two codebases with this name:

- **opencode-ai/opencode** (Go, archived Sept 2025): The original Go CLI. No fork flag.
- **sst/opencode** (TypeScript, active): The current maintained project. This is what users mean by "OpenCode" today. All findings below are from this repo.

---

## CLI Syntax

```bash
# Fork and continue the most recent session
opencode run "next message" --continue --fork

# Fork a specific session by ID
opencode run "next message" --session <session-id> --fork
```

The `--fork` flag is defined in `packages/opencode/src/cli/cmd/run.ts`:

```typescript
.option("fork", {
  describe: "fork the session before continuing (requires --continue or --session)",
  type: "boolean",
})
```

Guard in the handler:
```typescript
if (args.fork && !args.continue && !args.session) {
  UI.error("--fork requires --continue or --session")
  process.exit(1)
}
```

When `--fork` is present, the handler calls `sdk.session.fork({ sessionID: baseID })` with no `messageID`, which forks the entire session up to the latest message.

## What Gets Copied

The fork operation (`Session.fork` in `packages/opencode/src/session/index.ts`, lines 508-542) does:

1. Creates a new session via `createNext()` with the same `directory` and `workspaceID` as the original.
2. Loads all messages from the original session.
3. If a `messageID` cutoff is provided, stops before that message (exclusive: `msg.info.id >= input.messageID` breaks the loop). If no `messageID`, copies everything.
4. For each copied message:
   - Generates a new `MessageID` (ascending ULID).
   - Maintains parent-child relationships between assistant messages by remapping IDs through a local `idMap`.
   - Calls `updateMessage` with the new session ID and new message ID.
   - For each part of the message, calls `updatePart` with a new `PartID`, the new message ID, and new session ID.

**What is copied:** Every message (user and assistant), every part of every message (text, tool calls, file attachments, reasoning blocks, step markers — everything in the `PartTable`).

**What is NOT copied:** Todos (the `TodoTable` is session-scoped and not cloned). The original session's `revert`, `summary`, `share_url`, and `permission` fields are not transferred; the fork starts with those fields empty/default.

## Session Title

The fork title is generated by `getForkedTitle()`:

```typescript
function getForkedTitle(title: string): string {
  const match = title.match(/^(.+) \(fork #(\d+)\)$/)
  if (match) {
    const base = match[1]
    const num = parseInt(match[2], 10)
    return `${base} (fork #${num + 1})`
  }
  return `${title} (fork #1)`
}
```

So "Fix the login bug" becomes "Fix the login bug (fork #1)". Forking a fork increments the counter.

## Relationship Between Sessions

The `SessionTable` has a `parent_id` column. However, looking at the fork implementation, `createNext()` does **not** set `parent_id` from the original session. The `parent_id` field is used for a different purpose: tracking sub-agent task sessions (child agents spawned mid-session).

The server route `GET /:sessionID/children` retrieves sessions forked from a parent in the sub-agent sense, not the fork sense. **There is no stored parent-fork relationship.** A forked session is a fully independent entity. Deletion, archival, and listing treat it as any other root session.

The only connection between original and fork is the naming convention in the title.

## Storage

OpenCode uses SQLite with Drizzle ORM. The database path is:

```
$XDG_DATA_HOME/opencode/opencode.db   # typically ~/.local/share/opencode/opencode.db
```

Relevant tables (from `packages/opencode/src/session/session.sql.ts`):

- **`session`**: One row per session. Columns: `id`, `project_id`, `workspace_id`, `parent_id`, `slug`, `directory`, `title`, `version`, `share_url`, `summary_*`, `revert`, `permission`, `time_*`.
- **`message`**: One row per message. Columns: `id`, `session_id`, `time_created`, `data` (JSON blob containing role, model, parentID, etc.).
- **`part`**: One row per message part. Columns: `id`, `message_id`, `session_id`, `time_created`, `data` (JSON blob, polymorphic).

All data referenced in a session lives in these three tables. Fork is purely a database operation — no files are copied.

## UI-Level Fork (Point-in-Time)

The TUI and web app also expose fork-from-a-specific-message. The user sees a list of all user messages in the session and picks one. The fork includes all messages up to (but not including) that chosen user message, then navigates to the new session with that message pre-loaded in the prompt box. This uses the same `Session.fork({ sessionID, messageID })` API.

The web app dialog is at `packages/app/src/components/dialog-fork.tsx`.
The TUI dialog is at `packages/opencode/src/cli/cmd/tui/routes/session/dialog-fork-from-timeline.tsx`.

The CLI `--fork` flag (without messageID) forks the whole history and sends the new message to the fork.

## Summary

| Question | Answer |
|---|---|
| CLI flag | `opencode run <message> --continue --fork` or `--session <id> --fork` |
| What is copied | All messages and all message parts, up to optional messageID cutoff |
| Files/settings copied | Directory and workspaceID only; no todos, no sharing, no permissions |
| Storage | SQLite at `~/.local/share/opencode/opencode.db` |
| Original-fork relationship | None stored — fork is fully independent; linked only by title naming convention |
| Title | `"<original> (fork #N)"` |
| Implementation | Pure DB copy: new IDs for session/messages/parts, parent-child refs remapped |
