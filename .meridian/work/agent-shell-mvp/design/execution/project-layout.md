# Project Layout

> What this is: the normal local project shape the shell expects.
>
> What this is not: a special shell-owned workspace abstraction.

Back up to [overview.md](./overview.md).

## 1. Baseline

A user project stays a normal `uv`-managed Python project:

```text
project/
├── pyproject.toml
├── uv.lock
├── .venv/
├── .agents/          # generated by `meridian mars sync`
├── .meridian/        # state and work-item artifacts
└── src/ or notebooks or data/
```

The shell does not invent `/workspace`, a second package manager, or a second
virtualenv hierarchy for domain code.

## 2. Expected Workflow

1. `uv sync`
2. `meridian mars sync`
3. `meridian shell start`

The installed packages determine which agents, skills, MCP servers, and
interaction-layer extensions are available.

## 3. Work-Item Convention

The active work item still matters for files-as-authority, but it is an
ordinary directory under `.meridian/work/`. The shell and packaged MCPs read
and write there through normal filesystem paths.

## 4. Consequence

A package author who needs PyVista, SimpleITK, or another stack expresses that
through the package and its launch command, not through shell-owned global
bootstrap logic.
